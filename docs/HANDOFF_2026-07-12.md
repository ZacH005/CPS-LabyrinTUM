# CPS-ML-Maze — Status & Run Guide

**Snapshot: 2026-07-12** · branch `main` · repo `github.com/ZacH005/CPS-ML-Maze`

This is a handoff for continuing the autonomous marble-maze solver. It covers
**where we are**, the **one thing currently blocking us**, **how to run it**,
and the **gotchas** that will otherwise cost you hours.

## TL;DR

The software pipeline works end-to-end: camera → ball tracking → homography to
board-mm → path following → speed plan → servo commands → Arduino. The camera
and control loop were the big wins this week (see below). **The current blocker
is hardware, not software: the PITCH axis has mechanical backlash and must be
fixed before autonomous runs will work.** Everything downstream is ready and
waiting on a clean pitch axis.

## 🔴 THE BLOCKER: pitch-axis backlash (fix this first)

`axis_check.py` drives each servo **directly** (no controller) and measures how
far the ball rolls. Latest result:

| pulse | amplitude | ball moved |
|-------|-----------|------------|
| +yaw  | 0.40      | 77 mm ✅ crisp |
| −yaw  | 0.40      | 73 mm ✅ crisp |
| +pitch| 0.40 → 0.64 → 1.0 → 1.0 | ~1 mm each (nothing) |
| +pitch| 1.0 (5th try) | **108 mm** (sudden lurch) |
| −pitch| 0.40      | ~0 mm, then 101 mm at 0.64 |

Yaw moves cleanly at low tilt. **Pitch eats several near-max commands, then
lurches** — classic **backlash/slop in the pitch linkage**. The controller
sends normal pitch commands (0.1–0.3); on a loose linkage those produce
*nothing*, then occasionally break through and jerk. That is the "wiggles and
shakes but the ball won't move / freaks out at holes" behavior.

**No software or calibration can fix this.** To fix, physically:
1. Tighten the **pitch servo horn screw** (a loose horn slips on the spline).
2. Remove play in the **pitch tie-rod** (ball-joints + threaded rod).
3. Tilt the board by hand in pitch — there should be **no free play** before the
   board moves. Match the yaw side, which is tight.

Then **re-run `axis_check`** (ball in an open area). You want all four
directions to move cleanly at ~0.40 with similar magnitude. Only then will the
axis map be trustworthy — right now it is skewed (it boosted pitch gain to
~1.22 to compensate for the dead zone, which will over-drive pitch once the
linkage is fixed).

## What works now (recent wins)

- **Camera at 120 fps.** It was silently running at **10 fps** because OpenCV
  opened it via DirectShow (uncompressed YUY2, USB2 bandwidth-capped). We switch
  to **MSMF** on Windows (`src/cps_maze/camera.py`) → full **120 fps at
  1280×800**, no resolution change. This was the root cause of the old
  "can't stabilize" oscillation (a 10 Hz loop can't control a fast marble).
- **Control loop now ~62 Hz** (was 10.6 Hz).
- **Camera server** (`scripts/camera_server.py`): opens the camera once
  (MSMF is slow to open, ~10–30 s) and shares frames to every other script via
  shared memory, so calibration tools open **instantly**. Also stamps frames
  with true capture time, killing a phantom-velocity artifact.
- **Velocity estimator hardened** (`state_estimator.py`): ignores camera
  burst-frames (dt≈0) and clamps impossible jumps, so a detection glitch no
  longer reads as 400 mm/s and trips the safety logic.
- **Composure/stabilize state** (`run_autonomous.py`): when the ball genuinely
  runs away (>3× plan **and** ≥55 mm/s) it holds position and damps, then
  resumes — instead of pushing forward into a hole.
- **Wall-mask speed throttle is off-route only** — it was crushing on-route
  speed to 35 % and parking the ball at the start.
- **A\* recovery disabled by default** — the oblique camera view makes the ball
  look like it's touching walls, which fired constant false "wall recovery."
- **Neutral trim is currently 0** — we level the board by hand. (Local file,
  `calibration/neutral_trim.json`.)

## How to run (a normal lab session)

### 1. Start the camera server (once, leave it running)
```powershell
python scripts\camera_server.py
```
Wait for `serving ~120 fps`. Every other script now opens instantly. Ctrl-C to
release the camera at the end.

### 2. Recalibrate — only if the rig moved (camera/board/servos)
If nothing physical changed, skip this. If it did, calibration artifacts are
coupled (a new homography invalidates path/holes/wall-mask/ROI/axis-map).
Easiest is the orchestrator, which walks you through each step:
```powershell
python scripts\recalibrate_all.py
```
Or run the pieces individually (order matters):
```powershell
python scripts\calibrate_homography.py            # click the 4 play-area corners
python scripts\auto_detect_holes.py               # -> configs/maze_holes.csv
python scripts\auto_trace_path.py                 # -> path CSV
python scripts\build_wall_mask.py                 # -> calibration/wall_mask.npz
python scripts\select_maze_roi.py --capture       # -> calibration/live_roi.json
# confusers MUST come from a BALL-FREE clip (see gotchas):
python scripts\record_camera.py --duration-s 20   # ball OFF the board
python scripts\pipeline.py --calibrate data\raw\live_camera_<ts>.avi `
    --confusers-file calibration\live_confusers.json `
    --roi-file calibration\live_roi.json
# axis map — ball in an OPEN area, clear ~80 mm in all 4 directions:
python scripts\axis_check.py --port COM10 --map-mode normalized-response
```

### 3. Run autonomous
```powershell
python scripts\run_autonomous.py --port COM10
```
Useful flags: `--max-command 0.3` (gentle first test), `--dry-run` (no serial),
`--no-preview`, `--v-max 20`, `--max-seconds 60`. Logs to
`data/raw/autonomous_run.csv`. Click the ball to seed, then follow the arming
prompt.

### 4. Analyze the run
```powershell
python scripts\analyze_run.py                     # progress %, stalls, where it stuck
python scripts\visualize_run.py                   # annotated replay + "why" panel
python scripts\debug_tracking.py                  # live tracker diagnostics
```

## Gotchas (the stuff that will bite you)

- **Confusers must be recorded with NO ball on the board.** If the ball is in
  the calibration clip, the tracker blacklists the ball's own resting spots →
  random track losses. A healthy board has **~5–15 confusers**; if
  `pipeline.py --calibrate` reports dozens (we've seen 90–111), the clip had a
  ball — re-record and redo.
- **Axis check: put the ball in open space.** If the ball hits a wall mid-pulse,
  that direction reads short (e.g. −pitch = 36 vs 80) and silently skews the
  map. The tool now warns ("WEAK, likely hit a wall") — heed it.
- **Camera device index can shift** after a reboot/USB change. If the camera
  won't open or shows the wrong feed, fix `camera.device_index` in `local.yaml`.
- **Loop is 62 Hz now**, but the control gains were originally tuned at 10 Hz.
  If the ball oscillates *after* the pitch linkage is fixed and the axis map is
  clean, drop the velocity/pitch gain (`k_vel`, `kd`) — that's a loop-rate
  retune, not a recalibration.
- **Neutral trim is 0** — the board must be levelled by hand. Don't re-run
  `calibrate_neutral_trim.py` unless you want the software trim back.
- **Serial port** is held exclusively; if you get a COM10 PermissionError,
  another process (old run, Arduino IDE) has the port.

## How a frame becomes a servo command (architecture)

```
camera (MSMF 120fps) ─► camera_server (shared mem) ─► CameraCapture
   ─► PipelineBallTracker (motion + specular, ROI, confusers)   px
   ─► homography                                                board mm
   ─► path association (windowed + wall LOS)                    progress, cross-track
   ─► speed profile (precomputed) + carrot follower             desired velocity
   ─► composure / emergency / stall-kick guards                 board_cmd
   ─► axis_map (board dir → servo)                              yaw, pitch
   ─► neutral trim (currently 0)  ─► serial 500k ASCII "SET y p"
   ─► Arduino + PCA9685 ─► 2 servos tilt the board
```

Key modules: `scripts/run_autonomous.py` (the main loop),
`src/cps_maze/vision/` (tracking, homography, state estimator),
`src/cps_maze/planning/` (speed profile, holes, walls),
`src/cps_maze/control/` (followers, stall kicker, trim),
`src/cps_maze/hardware/serial_link.py` (Arduino link).
Tuning knobs are all in **`configs/default.yaml`** with inline comments.
